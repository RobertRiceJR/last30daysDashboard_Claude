Open Knowledge Format — should you care, and how much?

Google Cloud, v0.1 · Apache-2.0 · markdown + YAML frontmatter knowledge bundles for AI agents

Bottom line: OKF formalizes the exact pattern you already run by hand — markdown + YAML-frontmatter knowledge trees that agents read and write. Your knowledge store, your memory files, and your prod-support incident KB are proto-OKF today. That makes this less a "new tech to evaluate" and more a "standard that just validated your architecture."

Overall grade: B+ — exceptional on openness, simplicity, and strategic fit; weak on maturity, ecosystem, and governance (it's v0.1, Google-stewarded, no neutral standards body yet).

Recommendation: adopt the conventions, not the vendor. Align your existing markdown knowledge to OKF's shape (frontmatter schema, index.md, log.md, markdown-link graph) — it's cheap because you're 80% there — while staying independent of Google's Knowledge Catalog / ADK tooling until the ecosystem proves out.

What it is (in one breath)

An open spec that says: represent your organizational knowledge as plain markdown files with YAML frontmatter, shippable in git, linkable into a graph, readable by a human with cat and by an LLM verbatim. One required field (type); everything else optional. The spec "fits on a single page." Reference tooling (an enrichment agent over BigQuery, a static HTML graph visualizer) is explicitly proof-of-concept. Authored by Google Cloud's Data Cloud team; published Apache-2.0 on GitHub.

Why this maps onto what you already have

🗂️ Your knowledge store

_shared/knowledge/ is domain nodes in markdown with > Source: headers and See-also links — that's OKF concepts + frontmatter + a link-graph, minus the YAML shape. An OKF pass would mostly be formalizing headers into frontmatter.

🧠 Your memory files

Memory entries already carry name / description / metadata frontmatter + [[links]]. That is an OKF concept document in all but field names. Near-zero migration.

🛟 Prod-support incident KB

Asset-keyed markdown, appended incident history, a 00-index.md — OKF gives this an exact vocabulary: concepts + index.md (progressive disclosure) + log.md (change history).

📰 This research loop

Digests + synthesis sidecars could be emitted as OKF concepts, making each day's findings an agent-readable, linkable knowledge artifact rather than just HTML.

What to do

  • Adopt the conventions now (low-risk): standardize frontmatter across the knowledge store + memory toward OKF's shape; add index.md / log.md where you already have indexes and change logs. You gain interop optionality for ~free.
  • Stay tooling-independent: don't wire into Google's Knowledge Catalog / ADK to consume it — keep your own markdown pipeline. The format is the value; the vendor tooling is v0.1 POC.
  • Track maturation: watch for a neutral governance home and non-Google adopters before treating OKF as a stable dependency. (Now tracked in the research loop — see below.)
  • Don't rip-and-replace: this is alignment, not migration. Your architecture was right; OKF is the receipt.
